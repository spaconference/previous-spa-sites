SPA Conference session: Describing Information Systems: Moving Beyond UML

One-line description:A workshop that will attempt to identify the requirements and likely form of a visually oriented language for describing the architecture of information systems.
 
 Session materials
 
Session format: Workshop [read about the different session types]
 
Abstract:Part of the job of the software architect is to create a coherent and useful architectural description so that the architecture of their system can be communicated to and understood by interested stakeholders (including the architect himself).

UML is the defacto standard language for describing software models, butit is primarily an object-oriented software design notation and many architects find it difficult to express their ideas using it.

Alternatives to UML have been proposed in both the academic community (in the form of ADLs like ACME, xADL and Wright) and the industrial community(for example Gregor Holpe's "gregorgrams" notation used in his EAI patterns book). However, these alternatives have not been widely adopted by information systems practitioners.

This situation means that many architects end up inventing their own informal architectural design notations in order to describe their systems. This can be problematic for a number of reasons, not least the wasted effort involved in repeatedly reinventing notations, confusing inconsistency between architectural descriptions and the likelihood of confusion and misunderstanding when stakeholders need to understand and interpret new notations.

This workshop will attempt to identify the key requirements of a language for describing software architectures and consider how (or if) UML is lacking as a solution to them. Participants will then attempt to improve on UML by sketching their own visual language for defining software architectures.
 
Audience background:Architects, designers, developers and managers involved in the development of information systems. Ideally participants will have attempted to create at least one significant architectural description using UML or other graphical notations.
 
Benefits of participating:* An understanding of the requirements for an architectural description notation.
* Ideas as to how to improve their own architectural descriptions.
* The chance to start designing a new architectural notation.
 
Materials provided:* Presentation slides
* Worksheets for the exercises
 
Process:1. Introduce the objectives of the session and the process to be used.

2. Present a couple of examples of using UML for architectural description, pointing out how little information is really present in the standard notation. The presentation will then recap the importance of stakeholders in architectural design and the wide range of stakeholders that need to be served. We will also explain how a view-based approach to architectural description can help to communicate across a wide stakeholder community and suggest a standard set of views to consider (functional, concurrency, information, development, deployment and operational).

3. Split into groups and undertake first group exercise: "Who, What, How?" In this exercise the participants will consider which stakeholders need to understand which of the views in an architectural description, and what they need to understand from each. Then, for each view and stakeholder pair, they will identify the key requirement for communicating that part of the description to the stakeholder in question (e.g. "ease of understanding", "precision", "rapid assimilation", "overview", "details" and so on).

4. Collate the results of the first exercise using flip chart sheets on the walls.

5. Present a couple of slides containing our view of UML as an architectural description language (which is that it has its limitations, to say the least).

6. In the same groups, undertake the second exercise: "UML, Friend or Foe?" In this exercise, the groups will decide how well standard UML meets each of the requirements identified in the first exercise, by considering how effective it would be at communicating with each of the stakeholder groups considered in the first exercise.

7. Collate the results of the second exercise using flip charts on the walls.

8. Again in the same groups, undertake the third exercise "Quivering at Arrows". In this exercise, the groups will attempt to improve on UML by sketching a better architectural description language. This should identify the entities that can be represented, their semantics and relationships and the graphical notation to be used for each. Some example UML architectural description fragments will be provided to provide groups with concrete examples to work with.

9. Each group presents their language.

10. Summary and recap.
