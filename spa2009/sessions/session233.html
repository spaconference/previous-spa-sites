SPA Conference session: Can metrics increase quality in development teams?

One-line description:Implementing a 3 stage metrics process for development teams using various open source tools with proof of 'classic' metrics.
 
Session format: Tutorial (75 mins) [read about the different session types]
 
Abstract:Quality in Software Projects needs to start in the Development Team. Using tools and metrics can help spot buggy code and reduce risk of issues before the code has even entered QA. However, are all metrics equal? And what do we do with the data collected? This session aims to discuss and demonstrate that certain metrics used in a feedback loop such as Continuous Integration do benefit Software Projects and lower rework costs.
We also look at PROOF that combinations of metrics such as McCabe cyclomatic complexity, coupling, comment to code ratio etc can be predictors to spotting buggy code prior to deployment.
 
Audience background:Aimed for beginner/intermediate audience. All examples use Java but topic is largely language agnostic. Session is design for between 60-90 mins as a Technical Speaking engagement. Designed for Developers, team leaders and development managers.
 
Benefits of participating:Participants will learn:
• How to implement a 3-stage process using tools such as Static Code Analyzers, Code Coverage Tools and Dependency Analysis tools to provide feedback to the team so quality can be maintained through the coding phase, with examples of how these tools can be best used for the fastest return.
• Gaming metrics and how to spot people who do this as well as discussing how using metrics can psychologically increase or decrease the quality of the code (Hawthorn Effect).
• Proof that certain metrics such as McCabe Cyclomatic Complexity value does indicate buggy code based on real world, Open Source projects from Sourceforge.
• How to sell Quality initiative up to management.
