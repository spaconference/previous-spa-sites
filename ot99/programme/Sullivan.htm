XML Messaging at Chase Manhattan Bank Global Markets

A Case Study for OT99

John O'Sullivan

XML was developed as a web-friendly successor to SGML for generalised document content markup. However, its flexibility in modelling semantic content is leading to increased use as a more general medium for data exchange.

Chase Manhattan Global Markets has many applications deployed on its primary trading floors in New York and London which perform functions such as trade capture, position tracking, profit and loss calculation, value at risk calculation, and pricing for a wide range of wholesale financial products. These front office applications run on desktop systems. In the back office applications run on mainframes and minis and handle confirmation, settlements and payments for the trades done by the front office. File transfer and file sharing are the primary means of inter application messaging. Each application defines its own message formats; these formats specify messages as fixed length blocks of text, with the message fields occurring at fixed offsets and being of fixed widths.

Clearly the existing solutions to our application's messaging requirements are brittle and support intensive. To remedy this the Global Markets Technical Architecture Group are now building the ASAP Messaging System (AMS). AMS will connect applications running on a variety of platforms, and implemented in several programming languages. AMS is built on IBM's MQ series, which provides the guaranteed delivery and queueing across platforms that we need to make our inter application messaging more resilient. But MQ series says nothing about message formats; to MQ messages are just strings. So how should those strings be formatted?

Candidate message formats evaluated by the AMS team included XML, ASN.1 and a home brewed option. XML won out for several reasons. It's an industry standard supported by third party tools, that has backing from all the major vendors. It is human readable. It enables us to put semantics into messages, and so make them self describing. XML supports optional elements, and elements that may occur a varying number of times. This supports dissent between sending and receiving applications over message content. The flexibility of XML will enable many different Chase applications to agree on, for instance, a single foreign exchange trade message format that has many optional parts.

This case study includes a detailed discussion of the Content Model developed for AMS messages, the selection of a third party XML parser, and the message formatting tools that Chase has developed. Those tools translate Java and C++ objects, as well as C, COBOL and Natural data structures, to and from XML.

Who is this session for?

Architects investigating and defining messaging middleware standards for their organisations: developers faced with the problem of integrating disparate systems with complex information flows.

What will participants get from the session?

If previously unfamiliar with XML, participants will acquire a basic reading knowledge of XML. Delegates will come away with an understanding of the particular issues involved in using XML for messaging data transfer rather than generalised markup, in particular, the importance of a well-defined content model. They will also learn criteria upon which to base the selection of tools and technologies for XML.
