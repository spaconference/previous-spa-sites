A Case of Identity

Workshop

Object Identity in Architecture and Distribution

Charles Weir

Traditional OO programming depends strongly on the concept of an objectís identity; to invoke a message to an object one must have access to its identity. Different systems implement identity in different ways: as memory pointers or table indices or as file offsets, to name but three.

Distributed systems and OO databases keep the same OO concepts as the programming languages they reflect, including message invocation and identity. They must, to be easy for an OO programmer to use. However because they have a much more complicated environment, these concepts become strained. Message invocation is no longer simple ñ the system responsible for the remote object may not be there. And identity is no longer such a simple concept ñ for example:

  • If a process caches a local copy of an object, where is the 'real' object? When does the distinction matter?
  • How can we guarantee that two processes accessing the same object see the same thing?
  • What is the relationship between identity and transactions?
