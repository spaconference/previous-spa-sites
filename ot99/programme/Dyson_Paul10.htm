Programming in the Extreme

Monday 2:15   Workshop 170 minutes

A practical introduction to Extreme Programming

Paul Dyson

David Harvey

Extreme Programming (XP) is a software development discipline created by Kent Beck, Ward Cunningham and others, with much informative experience arising from the highly successful C3 (Chrysler Comprehensive Compensation) project at Chrysler Autos in the US. XP turns the traditional development methodologies on their heads: there is no documentation, code-level commenting is frowned on, no-one practices code ownership, there is very little in the way of explicit design, programmers and the practice of programming are aggressively promoted. In fact aggression (positive aggression directed at solving the problem in hand) is a theme: the XP promoters love to use bold statements like 'you ain't gonna need it' and 'the source code is the design'. The development process is highly iterative, the XP guidelines are lovingly refined and firmly enforced: either you're doing XP or you're not. This workshop will look at the XP as a whole, explore the underlying principles of the method and provide opportunities for practical experience of the XP style of software development.

Paul Dyson

Paul Dyson is currently with Coexis (formerly Cumulus Systems Ltd.) where he is part of a project developing a next-generation integrated front- and back-office system for the capital markets using Smalltalk, Java, CORBA ORBs and an ODBMS. Paul has presented sessions at a number of conferences (including previous OTs) on distributed architectures, architectural variation and style, software frameworks, and patterns.

Topics

    The workshop will cover the guiding principles of XP:

  • The four things that really matter
  • You ain't gonna need it
  • Do the simplest thing that could possibly work
  • Spike the solution
  • User stories
  • Test as specification
  • Refactoring and continuous integration
  • Pair programming
  • Code formatting
  • CRC cards for communication and design

Benefits

Participants will understand the motivations behind XP and the (perhaps paradoxical) discipline needed to ensure its effective application. XP captures the essential creativity of programming and provides an environment in which this can be harnessed by larger groups of developers: the session will provide both experience and food for thought on the nature of development, team interaction, and quality in software.

Level: intermediateAudience: Anyone who is or has been a developer. Anyone coming should be experienced with either Smalltalk, Java or both and be prepared to role their sleeves up and get coding.

    Format

    		 Two halves, 15 min break between them: First half: a number of (very) short presentations on the XP guidelines interspersed with short programming games. Some of these games are played in pairs as pen-and-paper exercises: others may involve a pair working at a computer with another pair observing (watching for evidence of XP, writing down XP idioms, recovering, etc.).

    Subjects for programming games might include: reversing a string, testing for the primality of a number, implementing a map preserving order of insertion, encoding/decoding strings, substring matching and replacing. Some of these will be set up to explore the dynamics of pair programming, some testing (for example, two teams each produce test code for theproblem: the test code is exchanged, the teams then implement the functionality to support the tests).

    Second half: a programming challenge designed to exercise the XP principles - groups have to build a system in very short time. The challenge comes in three stages:

    1. Familarisation and exploration. Each group is given a set of user stories. Time is given for them to read these and do some initial CRC scoping. All four group members work together.

    2. Implementation. One pair implements and one observes, with pairs swapping at regular intervals. Implementation is not only of the system but also of the unit tests and end-to-end tests. Observing pairs will be reflecting on the actions of the implementers and planning future actions.

    3. Modification. A number of new user stories are introduced. Groups have to make as many modifications as possible in a short time. Modifications will be designed to ensure refactoring is the key.

    A project for the coding challenge will be of a scope similar to the following:

    • Contact manager: holds names, addresses, appointments. Extensions include: optional fields or free-form notes, appointment alarms, to do list.
    • Resource assignment: usual teachers, qualifications, rooms problem. Extensions include: optimisation of resource allocation, new types of resource or qualification, what if scenarios.
    • Version control system: checks (virtual) files in and out. Extensions include: permissions on file check in/out, simple config maps, reporting.
    • Order routing: an order object is accepted by a system which determines whether to automatically accept, divert or block the order depending on the customer. Extensions include: testing on other properties of the order, discounts configurable for selected customers.
